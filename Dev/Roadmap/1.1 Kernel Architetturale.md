# Roadmap 1.1 Kernel Architetturale

Questa roadmap prepara la fase successiva alla 1.0 professionale. Non aggiunge nuove superfici utente come priorita iniziale: consolida kernel, pipeline, runtime e release per rendere il vault meno fragile mentre cresce verso Campaign OS avanzato, continuita narrativa e adapter meccanici modulari.

## Tesi

Il progetto ha gia superato la fase "vault avanzato". La prossima maturita non arriva aggiungendo altre dashboard, ma rendendo piu piccoli, testabili e riproducibili i punti centrali:

- build e release devono essere hermetic o quasi hermetic;
- runtime e adapter Obsidian devono essere separati dal dominio;
- i renderer dichiarativi devono condividere kernel comuni;
- continuita narrativa e regole D&D devono avere test comportamentali;
- plugin layer e release devono poter degradare senza rompere il primo percorso utente.

## Definition Of Done 1.1

- `npm run check` resta verde su repo sincronizzata e segnala chiaramente quando serve `npm run sync:sources`.
- Dipendenze Node, Python e SRD non dipendono da versioni implicite o da branch remoti mobili.
- I renderer cockpit usano un kernel condiviso invece di duplicare validazioni comuni.
- `z.engine/session_views.js` resta bridge pubblico, ma non cresce come registro manuale centrale.
- Almeno PG, continuita narrativa, release boundary e azioni Meta Bind critiche hanno test comportamentali puri o fixture dedicate.
- Il layer plugin distingue dipendenze core, supporto e opzionali con fallback verificabile.
- Le nuove funzioni di world simulation partono da stato, eventi e conseguenze, non da nuove viste decorative.

## Non Obiettivi

- Nuovi plugin.
- Nuova web app, backend o database esterno.
- Refactor estetici non collegati a check o riduzione del rischio.
- Riscrittura totale di `z.automazioni` o `z.engine`.
- Conversione immediata in plugin Obsidian dedicato.
- Aggiunta di lore, contenuti demo o materiale regolamentare non necessario al kernel.

## Guardrail Da Non Toccare

- YAML resta source of truth dichiarativa.
- Jinja resta layer di generazione dev/release, non runtime utente.
- Template Templater restano wrapper sottili.
- `release_boundary.yaml` resta il confine autoritativo tra repo e artefatto utente.
- `entity_model.yaml` e `validation_contract.yaml` restano il contratto semantico principale.
- `user_path.yaml` resta il vincolo sul primo percorso del DM non tecnico.
- SRD resta separato dal contenuto canonico del mondo.

## Fase 0 - Baseline E DX

Scopo: rendere esplicito lo stato da cui parte ogni lavoro.

Lavori:

1. Aggiungere un preflight chiaro per repo non sincronizzata: quando mancano output generati ignorati, il messaggio deve indicare `npm run sync:sources`.
2. Valutare se `npm run check` deve includere una modalita safe di sync preliminare o se deve restare strict.
3. Aggiungere lock o pin minimi per toolchain Node/Python.
4. Documentare in `Dev/Sviluppo Vault` la sequenza locale canonica: sync, check, release, clean.

DoD:

- un clone pulito produce errore azionabile, non una lista dispersiva di file mancanti;
- CI e sviluppo locale usano la stessa sequenza dichiarata;
- nessun generated file viene tracciato.

## Fase 1 - Build Riproducibile E SRD Hermetic

Scopo: ridurre il rischio release non riproducibile.

Lavori:

1. Pinnare le dipendenze Python (`pyyaml`, `jinja2`) e Node.
2. Eliminare dipendenza da branch remoto mobile per import SRD: usare commit pin, manifest con checksum o snapshot sorgente controllato.
3. Aggiungere check che fallisce se la sorgente SRD e `main` senza pin.
4. Separare import SRD di manutenzione da materializzazione release.

DoD:

- release e CI non cambiano comportamento per aggiornamenti esterni non dichiarati;
- `npm run release:clean` puo spiegare quale versione SRD sta includendo;
- `check:importers` copre pin/versione/checksum SRD.

## Fase 2 - Kernel Renderer Cockpit

Scopo: rimuovere duplicazione strutturale dai renderer cockpit.

Lavori:

1. Estrarre un modulo condiviso per validazione dashboard, target, fileclass, bases, queue, surface e generated release target.
2. Convertire progressivamente i renderer `render_*_cockpit.js` al kernel comune.
3. Tenere nei singoli renderer solo differenze di dominio: pannelli, code, target e payload specifici.
4. Aggiungere fixture minime per almeno tre cockpit rappresentativi: DM Dashboard, Living World, Import Maps.

DoD:

- una nuova regola cockpit si aggiunge in un punto centrale;
- i renderer specifici diventano piccoli e leggibili;
- `npm run check:source-pipeline` resta verde dopo ogni conversione.

## Fase 3 - Runtime Kernel E Bridge Pubblico

Scopo: evitare che `session_views.js` e `helpers.js` diventino colli di bottiglia permanenti.

Lavori:

1. Generare o dichiarare un manifest export per `z.engine`, mantenendo compatibili gli export pubblici esistenti.
2. Spostare registry, plugin symptoms e loader in moduli dedicati.
3. Separare helper puri da helper Obsidian adapter in `z.automazioni`.
4. Bloccare la crescita di `helpers.js`: ogni nuova famiglia deve avere modulo dedicato.

DoD:

- `session_views.js` resta bridge compatibile, non deposito di nuova logica;
- helper puri possono essere testati senza `app.vault`;
- runtime load check continua a coprire gli export pubblici.

## Fase 4 - Test Comportamentali Di Dominio

Scopo: passare da check statici a garanzie sul comportamento RPG.

Lavori:

1. Testare calcolo PG: caratteristiche, point-buy, competenze, HP e mapping SRD.
2. Testare continuita narrativa: player choice, conseguenza, propagazione, gap e recap player-safe.
3. Testare release boundary: copia, esclusioni, marker proibiti, moduli bridge richiesti.
4. Testare Meta Bind actions critiche con frontmatter mockato.

DoD:

- almeno una suite puo girare senza Obsidian aperto;
- i test catturano regressioni reali su dati YAML;
- `check:js` non e piu l'unica garanzia sugli script.

## Fase 5 - Plugin Layer A Profili

Scopo: ridurre rischio operativo da dipendenze plugin numerose.

Lavori:

1. Classificare plugin in `core`, `support`, `optional`, `maintenance`.
2. Rendere il primo percorso utente dipendente solo dal profilo core.
3. Verificare fallback Markdown per support/optional.
4. Allineare `plugin_matrix.yaml`, `plugin_contracts.yaml`, `plugin_bindings.yaml` e `release_boundary.yaml`.

DoD:

- release quality distingue errore bloccante da degrado accettabile;
- il DM puo avviare il vault anche con plugin non core da riattivare;
- la matrice plugin spiega impatto, fallback e check associato.

## Fase 6 - Continuita Narrativa Come Sistema

Scopo: trasformare la continuita narrativa da insieme di viste e azioni a modello stato/eventi.

Lavori:

1. Definire evento narrativo minimo: sorgente, target, causa, conseguenza, stato, visibilita player-safe.
2. Introdurre reducer o applicatore puro per conseguenze e propagazioni.
3. Collegare fazioni, missioni, tracciati e sessioni allo stesso contratto evento.
4. Aggiungere code operative per conseguenze non applicate e conflitti tra stati.

DoD:

- la propagazione non dipende solo da script UI;
- ogni conseguenza importante ha stato, origine e target verificabili;
- Worldbuilder Dashboard e Motore Mondo Vivo leggono lo stesso modello.

## Fase 7 - SRD E Adapter Meccanico Minimo

Scopo: chiarire cosa e compendio, cosa e regola e cosa e materiale al tavolo.

Lavori:

1. Separare dati SRD, opzioni PG e funzioni di calcolo in moduli distinti.
2. Aggiungere adapter ruleset per D&D 5.5-compatible senza legarlo ai soli wizard PG.
3. Formalizzare readiness meccanica per incontro, creatura, oggetto, ricompensa e sessione.
4. Evitare nuove regole hardcoded negli script: ogni opzione stabile nasce da YAML.

DoD:

- il PG wizard usa un rules adapter testabile;
- creature/incontri/oggetti hanno check meccanici coerenti con il mondo;
- aggiungere un nuovo ruleset futuro non richiede riscrivere il runtime principale.

## Ordine Consigliato

1. Fase 0: Baseline e DX.
2. Fase 1: Build riproducibile e SRD hermetic.
3. Fase 2: Kernel renderer cockpit.
4. Fase 4: Test comportamentali di dominio.
5. Fase 3: Runtime kernel e bridge pubblico.
6. Fase 5: Plugin layer a profili.
7. Fase 6: Continuita narrativa come sistema.
8. Fase 7: SRD e adapter meccanico minimo.

Il motivo dell'ordine e pratico: prima si riduce il rischio di build/release, poi si elimina duplicazione, poi si aggiungono test, infine si espande il dominio simulativo.

## Rischi Da Monitorare

- Troppe migrazioni contemporanee in `z.automazioni` possono rompere Templater e Meta Bind.
- Spezzare YAML senza compilazione o check puo peggiorare la tracciabilita invece di migliorarla.
- Rendere il plugin layer troppo dinamico puo nascondere errori reali di release.
- World simulation senza test produce automazioni opache e difficili da correggere.
- Un motore regole troppo ambizioso puo soffocare il valore principale: preparare e giocare sessioni migliori.

## Primo Blocco Eseguibile

Iniziare da un blocco piccolo e verificabile:

1. Aggiungere preflight/check DX per output generati mancanti.
2. Introdurre pin minimi toolchain Python/Node.
3. Aggiungere check contro SRD remoto non pinnato.
4. Estrarre il primo modulo condiviso cockpit senza convertire tutti i renderer.
5. Eseguire `npm run sync:sources` e `npm run check`.
