# Roadmap 1.0 Professionale

Questa e la roadmap attiva di chiusura release. Non e archivio storico: contiene solo criteri, gap e lavori che possono ancora cambiare la qualita del prodotto.

## Visione

Il vault deve essere un prodotto Obsidian AAA per DM e worldbuilder non tecnici: worldbuilding profondo, materiale giocabile, sessioni radicate nel mondo, conseguenze persistenti e vista giocatori sicura.

Il centro resta il Codex del mondo. D&D 5.5-compatible con SRD separato e il profilo meccanico principale per creature, incontri, oggetti, party e preparazione, ma non deve schiacciare il worldbuilding in un compendio.

## Stato Solido

- Pipeline dichiarativa: YAML TemplateFactory -> Jinja/Markdown -> runtime JS -> Obsidian.
- `npm run check` passa e copre template, workflow, runtime, plugin, release, demo contract, importers, repo hygiene e JS.
- Templater usa wrapper funzione in `z.automazioni/templater`, evitando export object non validi.
- Plugin e release sono governati da `Dev/TemplateFactory/modules/plugin_contracts.yaml`.
- Workflow principali e azioni rapide sono in `Dev/TemplateFactory/modules/workflows.yaml`.
- Le entita centrali hanno gate YAML di giocabilita: tavolo, movimento, conseguenza, collegamento.
- I template comuni chiedono scelta, posta, rischi, ricompense, uso al tavolo e prossima mossa.
- Release artifact e release quality riusano una sola build nei check, senza rumore inutile.

## Definition Of Done 1.0

- Un nuovo DM apre il vault e capisce cosa fare senza conoscere Obsidian.
- Le superfici principali non mostrano errori Templater, Meta Bind, Dataview, Tasks o plugin.
- Ogni flusso centrale ha fallback Markdown leggibile.
- Il worldbuilding produce scelta, pressione, conseguenza o materiale player-safe.
- La vista giocatori non espone campi DM.
- La release pulita si apre su `Inizia Qui` e resta usabile anche se un plugin va riattivato.
- La repo resta leggibile: niente file storici o test duplicati fuori contratto.

## Gap Bloccanti

1. **Audit visuale Obsidian**
   Aprire davvero le superfici principali e correggere solo errori visibili o frizione reale:
   `Inizia Qui`, `Setup Guidato`, `DM Dashboard`, `Worldbuilder Dashboard`, `Preparazione Sessione`, `Durante il Gioco`, `Cosa Succede Fuori Scena`, `Controllo Vault`, `Vista Giocatori`.

2. **Feedback giocabilita nelle dashboard**
   Portare il gate tavolo/movimento/conseguenza/collegamento dentro `Worldbuilder Dashboard` e `Controllo Vault`, in forma leggibile per un DM non tecnico.

3. **Taxonomy gate SRD/D&D**
   Rafforzare `taxonomy_depth.yaml`: classi, specie, background, talenti, incantesimi, regole, creature e oggetti devono dichiarare come entrano in mondo, campagna o sessione.

4. **Pulizia controllata `z.automazioni`**
   Ridurre i 99 script top-level solo se il movimento semplifica davvero sviluppo futuro e non rompe Templater, Meta Bind, TemplateFactory o release.

5. **Smoke release finale**
   Eseguire prova da utente finale su release pulita con plugin inclusi, senza assumere conoscenze tecniche.

## Fuori Scope 1.0

- Nuove dashboard decorative.
- Nuovi plugin.
- Web app, backend, database esterni o publish esterno come requisito di release.
- Lore proprietaria non autorizzata.
- Refactor estetici non verificabili.
- Documentazione lunga non collegata a un gate o a una superficie utente.

## Prossimo Blocco Consigliato

Implementare feedback di giocabilita nelle dashboard operative:

- leggere i gruppi del `playability_gate`;
- mostrare per ogni nota viva cosa manca tra tavolo, movimento, conseguenza e collegamento;
- collegare ogni mancanza a un'azione manuale o pulsante esistente;
- verificare con `npm run check`.
